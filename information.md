# 行程規劃器資料庫匯入教學

## 目的

這份文件的目標是先規劃「把目前前端行程資料改為存進資料庫，並且匯出時也從資料庫讀取」的做法。

目前專案的資料主體仍以瀏覽器本地儲存為主，下一步要做的是：

- 輸入行程時，寫入資料庫
- 讀取行程時，從資料庫查詢
- 匯出資料時，從資料庫產生匯出內容
- 讓資料與登入帳號綁定，提升安全性與可管理性

這份文件先講架構、流程、資料表設計與導入步驟，不直接修改網站程式。

---

## 目前狀態

目前專案已完成：

- 行程新增 / 編輯 / 刪除
- 多旅程管理
- 預算統計
- Google Maps 地點連結
- Supabase 登入流程

目前額外確認的新需求：

- 每個旅程不應固定只有 7 天
- 建議在建立 / 編輯旅程時，允許自訂旅程天數

目前資料來源仍偏向前端本地資料，因此還有這些限制：

- 換裝置會看不到原本資料
- 不同使用者資料尚未真正隔離在資料庫層
- 匯出內容不是來自後端資料主表
- 後續若要做權限控管、共享、審計紀錄會比較困難

---

## 建議的最終方向

建議使用 Supabase 當作資料庫與登入基礎，資料結構分成兩層：

1. `trips`
   - 一筆代表一個旅程
2. `trip_items`
   - 一筆代表旅程裡的一個行程項目

如果未來要做多人共享，再增加：

3. `trip_members`
   - 控制哪些人可看、可編輯哪一個旅程

---

## 建議資料表設計

### 1. `profiles`

用途：

- 對應登入使用者
- 保存角色資訊

目前你已經有這張表，後續可繼續沿用。

建議欄位：

- `id`
- `username`
- `role`
- `created_at`

---

### 2. `trips`

用途：

- 存每一個旅程主檔

建議欄位：

- `id`
- `user_id`
- `name`
- `start_date`
- `duration_days`
- `show_all`
- `created_at`
- `updated_at`

欄位說明：

- `user_id`：這筆旅程屬於哪個登入者
- `name`：旅程名稱，例如「東京出差 4 月」
- `start_date`：旅程起始日
- `duration_days`：旅程總天數，供看板動態產生天數
- `show_all`：是否顯示所有日期

建議限制：

- 最少 `1`
- 最多 `40`

原因：

- 避免使用者輸入極大天數，造成畫面難以操作

---

### 3. `trip_items`

用途：

- 存旅程中的每一筆行程

建議欄位：

- `id`
- `trip_id`
- `title`
- `location`
- `transport`
- `budget`
- `start_local`
- `notes_html`
- `created_at`
- `updated_at`

欄位說明：

- `trip_id`：關聯到哪一個旅程
- `start_local`：行程日期時間
- `budget`：單筆預算
- `notes_html`：備註內容

---

## 資料關聯圖

- 一個使用者 `profiles`
- 可以擁有多個 `trips`
- 一個 `trip`
- 可以擁有多個 `trip_items`

也就是：

- `profiles 1 -> many trips`
- `trips 1 -> many trip_items`

---

## 權限與安全原則

這一段很重要，因為你的目標是前後端安全。

### 必須遵守

- 前端只使用 `anon key`
- 前端絕對不能放 `service_role key`
- 所有資料表都要開啟 RLS
- 使用者只能讀寫自己的旅程與旅程項目

### 權限控制建議

`trips`

- 只能讀自己的資料
- 只能新增自己的資料
- 只能更新自己的資料
- 只能刪除自己的資料

`trip_items`

- 只能操作屬於自己旅程底下的資料

換句話說，`trip_items` 的權限不能只看自己，而要透過 `trip_id -> trips.user_id` 去判斷。

---

## 導入方式建議

不要一次把全部功能硬切到資料庫，建議分三階段。

### 階段 1：資料表先建立

先完成：

- `trips`
- `trip_items`
- 權限政策
- 關聯設計

這一階段先不改網站邏輯，只把後端地基打好。

---

### 階段 2：先做「匯入資料庫」

這一階段目標：

- 保留原本前端操作體驗
- 當使用者新增 / 編輯 / 刪除行程時，同步寫入資料庫

建議策略：

- 一開始先做單向同步：前端 -> 資料庫
- 本地資料先保留作為備援
- 發生錯誤時，畫面要提示「資料庫同步失敗」

這樣可以先確認：

- 資料表設計對不對
- 權限政策對不對
- 登入使用者能不能正常存取自己的資料

---

### 階段 3：改成「資料庫為主」

這一階段目標：

- 頁面載入時直接從資料庫讀
- 旅程切換時直接查資料庫
- 匯出 JSON 時，從資料庫整理後輸出

完成後，本地儲存只保留：

- 暫存
- 離線備援
- UI 設定

真正的主資料來源改成資料庫。

補充：

- 之後看板不再固定 7 天，而是依 `duration_days` 動態產生

---

## 匯入資料庫的建議流程

### 做法 A：首次登入時匯入

適合現況：

- 使用者之前已經在本地建立很多資料
- 第一次改版後，要把舊資料搬到資料庫

流程：

1. 使用者登入
2. 系統檢查資料庫是否已有旅程
3. 如果沒有，再檢查瀏覽器本地是否有舊資料
4. 若有，就提示使用者是否匯入
5. 將本地多旅程資料寫入 `trips` 與 `trip_items`
6. 匯入成功後，再把資料庫當主要來源

優點：

- 對既有使用者最友善
- 不會一上線就讓舊資料消失

---

### 做法 B：提供手動匯入

流程：

1. 使用者從舊版匯出 JSON
2. 到新版登入後點「匯入」
3. 後端解析 JSON
4. 寫入 `trips` 與 `trip_items`

優點：

- 較好除錯
- 可重複操作

建議最終兩種都保留。

---

## 從資料庫匯出的建議流程

未來匯出資料時，不應再只匯出前端記憶體中的內容，應改成：

1. 取得目前登入者
2. 查詢該使用者可存取的 `trips`
3. 查詢每個 `trip` 對應的 `trip_items`
4. 組成目前前端可讀的 JSON 結構
5. 下載為檔案

優點：

- 匯出內容可保證與後端資料一致
- 即使換裝置，也能拿到同樣資料

---

## 建議的 JSON 匯出結構

建議維持和目前前端接近的格式，避免前端改太多：

- `version`
- `exportedAt`
- `data`
  - `activeTripId`
  - `ui`
  - `trips`
    - `id`
    - `name`
    - `startDate`
    - `durationDays`
    - `items`

這樣你未來即使從資料庫讀，也能維持原本匯出 / 匯入格式一致。

## 資料庫實作順序建議

### 第一步

先在 Supabase 建立：

- `trips`
- `trip_items`
- foreign key
- RLS

### 第二步

先做基本 CRUD：

- 新增旅程
- 新增行程
- 更新行程
- 刪除行程

### 第三步

把頁面初始化改為：

- 先檢查登入
- 再查資料庫
- 最後依 `start_date + duration_days` 渲染看板

### 第四步

改寫匯出邏輯：

- 匯出來自資料庫查詢

### 第五步

保留本地資料作為：

- migration 用
- 失敗備援用

## 導入前你需要先準備什麼

你在開始寫程式前，建議先確認這幾件事：

- Supabase Auth 已可正常登入
- `profiles` 表與 RLS 已正常
- 已能分辨 `admin` / `user`
- 已確認目前前端資料格式
- 已確認旅程天數要不要固定 7 天（目前決議：改為旅程可自訂）
- 已決定要不要支援舊資料自動搬移

---

## 風險提醒

### 1. 直接全改成資料庫，舊資料可能遺失

建議先保留本地資料，等匯入機制穩定後再切主來源。

### 2. `trip_items` 權限容易寫錯

最常見問題是：

- 使用者可以看到別人的行程
- 或是明明自己的旅程卻不能修改

所以這一段 RLS 要特別仔細。

### 3. 匯出格式不能隨便改

如果未來資料庫匯出格式和現在差太多，前端匯入邏輯會一起變複雜。

### 4. 天數改為可自訂後，要注意 UI 與資料同步

如果旅程天數改成可變動：

- 前端要改成動態產生天數欄位
- 匯入 / 匯出格式要保存 `durationDays`
- 資料庫 `trips` 要保存 `duration_days`

## 目前結論

你要的方向是正確的：

- 前端只負責畫面與操作
- 真正資料放資料庫
- 匯出以資料庫查詢為準
- 用登入身份做資料隔離
- 旅程顯示天數改由資料欄位決定，而不是前端寫死

這樣之後才能繼續往下做：

- 多裝置同步
- 多人共享旅程
- 管理後台
- 更完整的權限控制
